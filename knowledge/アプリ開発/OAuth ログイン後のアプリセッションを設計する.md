# OAuth ログイン後のアプリセッションを設計する

OAuth プロバイダーから利用者情報を取得した後も、アプリ内ユーザーの識別、ログイン状態の維持、アクセス制御はアプリ側で設計する。OAuth のアクセストークンを、そのままアプリのセッションIDとして扱わない。

## ログイン時の流れ

1. OAuth プロバイダーから取得した不変な利用者識別子を検証する。
2. プロバイダー名と利用者識別子の組み合わせを、アプリ内ユーザーへ対応付ける。
3. 推測困難なセッションIDを発行し、利用者ID、有効期限などをサーバー側へ保存する。
4. セッションIDを Cookie でブラウザーへ返す。
5. 保護対象ルートではセッションを検証し、必要な権限も確認する。

メールアドレスは変更される可能性があるため、それだけを外部アカウントの恒久的な主キーにしない。複数の OAuth プロバイダーを扱う場合は、識別子の衝突を避けるためプロバイダー名も保存する。

## Cookie とトークンの扱い

- セッション Cookie には `Secure`、`HttpOnly`、用途に合う `SameSite` 属性を設定する。
- セッションには有効期限を設け、必要に応じて更新時にもIDを再発行する。
- OAuth のアクセストークンをブラウザーへそのまま返さない。
- 外部 API の利用にトークンが必要な場合だけ、暗号化またはアクセス制御されたサーバー側ストレージへ保存する。
- OAuth のクライアントシークレットやトークンをログへ出力しない。

## ログアウトと失敗時の処理

ログアウト時は、Cookie を消すだけでなくサーバー側のセッションも無効化する。Google API など外部サービスとの接続解除も提供する場合は、アプリのログアウトとは分けて、必要に応じて OAuth トークンを失効させる。

利用者による同意拒否、OAuth コールバックのエラー、期限切れセッション、無効なセッションIDを通常の分岐として処理し、再ログインできる導線を用意する。
